# 文件区快速搜索（键入唤起）实施评估与分步方案

## 需求目标（整理版）

当焦点在文件列表区域时：

1. 用户按下任意字母键，可连续输入关键词。
2. 在列表容器底部弹出一个快速搜索工具条，显示当前输入内容。
3. 文件名中匹配关键词的片段高亮。
4. 列表自动定位到第一个匹配文件（滚动并可选中，至少保证可见）。
5. 工具条右侧有关闭按钮，点击后关闭工具条并清空关键词。
6. 工具条失去焦点后，5 秒自动关闭并清空关键词。

## 难度评估

- **总体难度：中等偏上（约 4~6 人日）**
- **是否建议分步：建议，且有必要**

原因：

1. **基础能力可复用，但行为链路较长**
   - 已有名称高亮能力：`FileListTextHighlight.attributedName(...)`。
   - 已有表格键盘入口：`FileListTableView.keyDown(...)` -> `FileListTableController.handleKeyDown(...)`。
   - 现有 `searchText` 语义偏“过滤列表”，新需求是“不过滤、仅高亮+定位”，需要并行状态。

2. **涉及 AppKit + SwiftUI 跨层状态同步**
   - 键盘事件在 AppKit 表格层。
   - 工具条 UI 在 SwiftUI 容器层更易实现。
   - 需要新增状态桥接与事件回调，避免耦合混乱。

3. **焦点与计时器是主要风险点**
   - “失焦 5 秒后自动消失”需正确区分：列表焦点、工具条焦点、窗口激活状态。
   - 键入期间/重新聚焦时需要重置倒计时，避免误收起。

## 建议分步执行

## Phase 1：打通最小闭环（键入 -> 工具条 -> 高亮 -> 定位）

### 目标

先实现核心交互，不含“5 秒失焦自动消失”。

### 主要改动点

1. **新增快速搜索状态（与现有 `searchText` 分离）**
   - 在 `ContentView` 增加：
     - `quickSearchText`
     - `isQuickSearchVisible`
   - 保留原有顶栏搜索 `searchText` 逻辑，不互相覆盖。

2. **扩展表格交互回调**
   - 在 `FileListTableInteraction` 新增回调：
     - `onQuickSearchInput(String)`：追加字符
     - `onQuickSearchBackspace()`
     - `onQuickSearchEscape()`
     - `quickSearchText`（用于高亮）
   - `searchText` 继续作为现有过滤用途，快速搜索单独走 `quickSearchText`。

3. **在表格键盘入口拦截字母输入**
   - `FileListTableController.handleKeyDown(_:)` 中新增逻辑：
     - 当焦点在表格且非 `command/control/option` 组合键时：
       - 字母键：触发 `onQuickSearchInput`
       - 删除键：触发 `onQuickSearchBackspace`
       - `Esc`：触发 `onQuickSearchEscape`
   - 对现有 Delete 文件删除逻辑做优先级约束：快速搜索可见时，删除键优先编辑关键词。

4. **文件名高亮切换为快速搜索词**
   - `FileListTableController.applyNameLabel(...)` 里用于高亮的文本来源改为：
     - 若 `quickSearchText` 非空：用 `quickSearchText`
     - 否则：退回 `interaction.searchText`

5. **首个匹配项定位**
   - 在 `FileListTableController` 增加：
     - 根据 `quickSearchText` 计算 `displayRows` 第一个匹配索引。
     - 调用 `scrollRowToVisible(index)`。
     - 可选：同步选中该行（建议开关可控，首版可先只滚动）。

6. **底部工具条 UI（SwiftUI）**
   - 在文件列表容器（`ExplorerFileListSection` / `fileTable` 外层）增加底部 overlay：
     - 左侧显示 `quickSearchText`
     - 右侧关闭按钮（X）
   - 点击关闭时清空快速搜索状态。

### 验收标准

- 焦点在文件区，按字母可连续输入并显示工具条。
- 文件名匹配片段可高亮。
- 每次输入后滚动到首个匹配项。
- 点关闭按钮后：工具条消失 + 高亮清除 + 定位效果释放。

---

## Phase 2：失焦 5 秒自动消失（焦点管理与计时器）

### 目标

补齐“快速搜索框失去焦点 5 秒自动关闭”行为。

### 主要改动点

1. **定义“失焦”语义**
   - 推荐语义：工具条可见时，只要焦点不在“表格或工具条”任一处，即视为失焦。

2. **新增自动关闭定时器**
   - 在 `ContentView` 增加 `DispatchWorkItem?` 或 `Timer`：
     - 失焦时启动 5 秒计时。
     - 重新聚焦或继续键入时取消计时。
     - 计时到点执行关闭并清空关键词。

3. **接入焦点事件**
   - 表格获得焦点、工具条文本框获得焦点、窗口失活等事件都要联动计时器。
   - 防止窗口切换回来后状态错乱（例如误保留旧关键词）。

### 验收标准

- 工具条失焦后 5 秒自动关闭。
- 5 秒内恢复焦点会取消自动关闭。
- 连续切换焦点不会出现工具条闪烁或误消失。

## Phase 3：可用性与边界完善（建议）

### 目标

降低误触，完善细节和回归稳定性。

### 可选增强

1. **输入范围控制**
   - 仅接收字母/数字/常见符号；过滤功能键与输入法组合态。

2. **无匹配反馈**
   - 工具条显示“无匹配”，并保持当前列表位置不抖动。

3. **快捷键一致性**
   - `Enter`：打开首个匹配项（可选）。
   - `Cmd+F`：聚焦现有顶部搜索（与快速搜索职责区分）。

4. **回归测试补充**
   - 键盘输入、关闭按钮、失焦计时、删除键优先级、拖拽与右键菜单互不影响。

### 验收标准

- 常见边界行为符合预期，无明显交互冲突。

## 风险清单与规避建议

1. **与现有顶栏搜索冲突**
   - 规避：状态分离（`searchText` vs `quickSearchText`），避免复用同一字段。

2. **Delete 键语义冲突（删文件 vs 删关键词）**
   - 规避：快速搜索可见时优先编辑关键词；为空时再走删文件逻辑。

3. **焦点事件不稳定导致计时误触发**
   - 规避：统一封装“进入/离开可交互焦点域”状态机，避免散落判断。

4. **高亮刷新性能**
   - 规避：沿用现有 `refreshVisibleNameLabels()`，仅刷新可见行。

## 结论

- 这项功能**不是高风险重构**，但涉及键盘事件、焦点、跨层状态同步，直接一次性做完容易引入回归。
- **建议按 2 个必做阶段（Phase 1 + Phase 2）推进**，Phase 3 作为稳定性增强按时间插入。
- 若要尽快上线，建议先交付 Phase 1（核心可用），再在下一迭代补齐 5 秒失焦自动关闭。
